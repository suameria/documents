# テストポリシー

- 単体テスト

- 結合テスト

```text
テストを楽に行うため基本的にはDIでの実装を行う！
(メソッド内でのnewはなるべく禁止)

@todo ↑　なんでDIでnewはだめなの？説明追加する

このテストポリシーはLaravelを前提条件として記載している。

自動テストに用いるライブラリはPHPUnitを使用する。

自動テストを用いることで何度もテストをすることが可能。

デグレの早期発見につながる。

データベースはテスト用のデータベースとテスト用エンドポイントを用意すること。

DynamoDBもテスト用エンドポイントを用意すること。

Redisもテスト用エンドポイントを用意すること。

外部サービスを利用する場合はモックを必ず利用すること。

Xdebugを利用してカバレッジの計測を行う。
```

## 単体テスト

```text
作成した全クラスの全メソッドに対してテストを行う。

[Unit/Service]
ビジネスロジック内でどのようなクラスが何回呼ばれているのか検証を行う。
テスト対象メソッド内で呼んでいるメソッドはモックを利用する。

[Unit/Repository]
データ操作ではfactory()でデータを投入して参照、登録、更新、削除の検証を行う。
```

## 結合テスト

```text
[Feature(共通)]
HTTPテストを行う。
リクエストのバリデーション、ビジネスロジック、データ操作での結合テストになる。
必要なデータがある場合は予めfactoryで生成しておくこと。
ヘッダー指定がある場合はwithHeadersを用いる。
200テストから検証する。
400テストを検証してみる。
サービスクラスが呼ばれる前のリクエストクラスの検証はデータプロバイダーを利用したほうが楽。
```

```text
[Feature(API)]
データ構造を確認してみる。
```
